# What is Amazon CloudWatch?

Amazon CloudWatch is a fully managed monitoring and observability service provided by AWS.

It collects operational data from AWS resources and applications, allowing you to monitor performance, analyze logs, visualize metrics, and respond to events.

CloudWatch enables you to proactively detect issues before they impact your applications.

It supports monitoring for:

- Amazon EC2
- Amazon EBS
- Amazon EFS
- Amazon RDS
- AWS Lambda
- Amazon S3
- Custom applications
- Operating system metrics
- Application logs

CloudWatch integrates with many AWS services, making it the central monitoring platform for AWS environments.

Amazon CloudWatch helps us monitor:

- CPU Utilization

# Core Components of Amazon CloudWatch

Amazon CloudWatch consists of several components that work together.

| Component | Purpose |
|-----------|---------|
| Metrics | Numerical performance data collected over time |
| Logs | Collect and store application and system logs |
| Dashboards | Visualize metrics using customizable widgets |
| Alarms | Trigger actions based on metric thresholds |
| Events / EventBridge | Respond automatically to AWS events |
| CloudWatch Agent | Collect additional operating system metrics and logs |

---

# Metrics vs Logs vs Alarms

Understanding these concepts is essential when working with CloudWatch.

## Metrics

Metrics are numerical values collected over time.

Examples:

- CPU Utilization
- Network In
- Network Out
- Disk Read Bytes
- Disk Write Bytes

---

## Logs

Logs contain detailed event information generated by applications or operating systems.

Examples:

- Apache Access Logs
- Apache Error Logs
- System Logs
- Authentication Logs
- Application Logs

---

## Alarms

CloudWatch Alarms continuously evaluate metrics.

When a threshold is exceeded, CloudWatch can:

- Send an email notification
- Trigger an SNS topic
- Invoke an AWS Lambda function
- Perform Auto Scaling actions

In the next document, we will configure CloudWatch Alarms for our monitoring server.

# Step 1: Create an IAM Role for CloudWatch

The CloudWatch Agent requires permission to send metrics and log data to Amazon CloudWatch.

## Create the IAM Role

1. Open the **AWS Management Console**.
2. Search for **IAM**.
3. Select **Roles**.
4. Click **Create role**.

<p align="center">
  <img src="../screenshots/CloudWatch/01-create-role.png" width="900" alt="Create IAM Role">
</p>

---

## Configure the Role

Select the following options:

| Setting | Value |
|----------|-------|
| Trusted Entity | AWS Service |
| Service | EC2 |

Click **Next**.

Search and attach the following policy:

```
CloudWatchAgentServerPolicy
```

Optionally attach:

```
AmazonSSMManagedInstanceCore
```

This policy allows Systems Manager to manage the CloudWatch Agent remotely.

Give the role an appropriate name.

Example:

```
EC2-CloudWatch-Agent-Role
```

Click **Create role**.

# Step 5: Create the CloudWatch Agent Configuration

AWS provides a configuration wizard that generates the agent configuration file.

Run:

```bash
sudo /opt/aws/amazon-cloudwatch-agent/bin/amazon-cloudwatch-agent-config-wizard
```

The wizard will ask a series of questions about:

- Operating System
- Metrics Collection
- Log Collection
- CloudWatch Namespace
- Log Files
- Disk Monitoring
- Memory Monitoring

After completing the wizard, a configuration file is generated.

Default location:

```text
/opt/aws/amazon-cloudwatch-agent/etc/amazon-cloudwatch-agent.json
```

<p align="center">
  <img src="../screenshots/CloudWatch/06-config-wizard.png" width="900" alt="CloudWatch Agent Configuration Wizard">
</p>

---

# Step 6: Start the CloudWatch Agent

Start the CloudWatch Agent using the generated configuration.

```bash
sudo /opt/aws/amazon-cloudwatch-agent/bin/amazon-cloudwatch-agent-ctl \
-a fetch-config \
-m ec2 \
-c file:/opt/aws/amazon-cloudwatch-agent/etc/amazon-cloudwatch-agent.json \
-s
```

If the command completes successfully, the CloudWatch Agent will begin sending metrics and logs to Amazon CloudWatch.

# Step 11: Configure Log Collection

Amazon CloudWatch can also collect log files from the EC2 instance.

Typical log files include:

| Log File | Purpose |
|-----------|---------|
| /var/log/messages | System messages |
| /var/log/secure | Authentication logs |
| /var/log/httpd/access_log | Apache access logs |
| /var/log/httpd/error_log | Apache error logs |

These logs are configured in the CloudWatch Agent configuration file.

Example:

```json
{
  "logs": {
    "logs_collected": {
      "files": {
        "collect_list": [
          {
            "file_path": "/var/log/messages",
            "log_group_name": "SystemLogs"
          }
        ]
      }
    }
  }
}
```

# Best Practices

Follow these recommendations when using Amazon CloudWatch in production environments.

### Monitoring

- Monitor CPU, memory, disk, and network utilization.
- Collect application logs and operating system logs.
- Create dashboards for critical infrastructure.
- Review metrics regularly.

### Security

- Use IAM roles instead of access keys.
- Restrict permissions using the principle of least privilege.
- Enable encryption where applicable.

### Cost Optimization

- Collect only the metrics you need.
- Remove unused dashboards.
- Define appropriate log retention policies.
- Delete unused Log Groups.

### Reliability

- Configure CloudWatch Alarms for critical metrics.
- Test monitoring after every deployment.
- Verify the CloudWatch Agent status regularly.

The following examples demonstrate how Amazon CloudWatch is commonly used in production environments.

A web application suddenly experiences a spike in user traffic during a promotional event.

CloudWatch continuously monitors CPU utilization.

An application generates log files every day.

Over time, the root volume begins to fill up.

CloudWatch monitors disk utilization and displays the increasing usage on the dashboard.

Combined with CloudWatch Alarms and Amazon SNS, administrators can receive email notifications immediately.

# Troubleshooting

Even with a correct setup, you may encounter issues while configuring Amazon CloudWatch. The following table lists common problems, possible causes, and recommended solutions.

| Issue | Possible Cause | Solution |
|-------|----------------|----------|
| CloudWatch Agent is not running | Agent service stopped | Restart the service using `systemctl` |
| No metrics appear in CloudWatch | IAM Role missing or incorrect | Attach the `CloudWatchAgentServerPolicy` policy to the EC2 IAM Role |
| Memory or Disk metrics are missing | CloudWatch Agent configuration does not include these metrics | Update `cloudwatch/cloudwatch-agent-config.json` and restart the agent |
| Log Groups are not created | Incorrect log file path | Verify the file path exists and the configuration file is correct |
| Permission denied while reading logs | Insufficient file permissions | Grant the CloudWatch Agent permission to read the log files |
| Dashboard widgets show "No Data" | Metrics are not yet available | Wait a few minutes and refresh the dashboard |

# Interview Questions

## 1. What is Amazon CloudWatch?

**Answer:**

Amazon CloudWatch is a monitoring and observability service that collects metrics, logs, and events from AWS resources and applications. It helps administrators monitor system health, troubleshoot issues, and automate responses using alarms.

---

## 2. Why do we need the CloudWatch Agent?

**Answer:**

The default EC2 metrics include CPU, Network, and Disk I/O. The CloudWatch Agent extends monitoring by collecting operating system metrics such as memory usage, disk utilization, swap usage, and application logs.

---

## 3. What is the difference between Metrics and Logs?

| Metrics | Logs |
|----------|------|
| Numerical performance data | Detailed event records |
| Used for dashboards and alarms | Used for troubleshooting |
| Lightweight | More detailed |

---

## 4. What happens if the CloudWatch Agent stops?

**Answer:**

The EC2 instance continues running normally, but new system metrics and log data are no longer sent to CloudWatch. Existing historical data remains available for analysis.

---

## 5. How does CloudWatch improve production operations?

**Answer:**

CloudWatch provides centralized monitoring, dashboards, and log collection. Combined with CloudWatch Alarms and Amazon SNS, it enables operations teams to detect issues early, respond quickly, and reduce downtime.
